# Spaced Repetition Flashcard App — Constitution

## Core Principles

### I. Tech Stack & Architecture Rules

The project MUST use Next.js (App Router only) with TypeScript, Supabase as the database/auth layer, and Tailwind CSS for styling. Server Components and Next.js Server Actions MUST be the default for logic-heavy operations. Client Components (`"use client"`) are permissible ONLY when browser-side state management or interactivity is genuinely required (e.g., card flipping, swipe gestures, animations).

**Why this matters**: This stack ensures type safety, scalable architecture, and clear separation of concerns. Server-first patterns reduce client bundle size and attack surface. Strictness on component placement prevents accidental performance regressions in a learning app where performance is essential.

### II. Code Quality & TypeScript Conventions

TypeScript strict mode MUST be enabled globally. The `any` type is forbidden except under extraordinary circumstances; use `unknown` with proper type narrowing instead. All function signatures, props, return types, and database schemas MUST have explicit type definitions. Store type definitions in `@/types`. Use English (camelCase for variables/functions, PascalCase for components) and apply Single Responsibility Principle strictly.

**Why this matters**: Type safety catches bugs at compile time. Clear naming in English ensures maintainability. Centralized types prevent duplication and reduce cognitive load during code review.

### III. Supabase & Database Rules

Every table MUST have Row Level Security (RLS) enabled with corresponding policies (Select, Insert, Update, Delete) tightly scoped to `auth.uid()`. Public access to tables containing personal user data is prohibited. MUST use the `Database` type generated by Supabase (via `supabase-js`) rather than manual type definitions. Instantiate Supabase clients using `@supabase/ssr` for correct behavior across Server Components, Client Components, and Route Handlers.

**Why this matters**: RLS is your primary defense against data leakage and privilege escalation. Type-generated schemas stay in sync with migrations. SSR client setup ensures proper context isolation and function availability across runtime boundaries.

### IV. Edge Runtime & Deployment Constraints

All Server Actions and API Route Handlers that implement SM-2 spaced repetition calculations MUST run on Edge Runtime (`export const runtime = 'edge'`). Avoid all native Node.js modules (e.g., `fs`, `path`, `crypto` Node variant) in Edge-targeted files; use Web APIs instead (e.g., `crypto.subtle`).

**Why this matters**: Edge Runtime deployment provides low latency for SM-2 computations worldwide. Web API compatibility ensures the code remains portable and testable across runtimes (Cloudflare Workers, Vercel Edge Functions, local Edge MockWorker).

### V. UI/UX & Styling Standards

Styling MUST use Tailwind CSS utilities exclusively; inline styles are prohibited. For complex conditional styling, use utilities like `clsx` or `tailwind-merge` for clarity. Animations—especially card flips, swipes, and transitions—MUST use Framer Motion to maintain consistency and simplify maintenance. Development MUST prioritize mobile-first design; desktop/tablet rendering follows as incremental improvements.

**Why this matters**: Flashcard learning happens predominantly on mobile devices. Tailwind prevents style drift and keeps CSS predictable. Framer Motion centralizes animation logic, reducing bugs and making timing changes straightforward.

### VI. Error Handling & Security

All Server Actions and database queries MUST be wrapped in `try/catch` blocks. Input validation is mandatory—use Zod (or equivalent) to validate all user inputs before database operations or SM-2 calculations. Validation failures MUST result in descriptive error messages and MUST NOT expose internal details.

**Why this matters**: Layered error handling prevents app crashes and data corruption. Input validation is your first line of defense against injection attacks, malformed data, and unexpected edge cases. Descriptive errors aid debugging; hidden details protect security.

## Development Workflow & Quality Gates

- **Pull Requests**: All PRs MUST reference compliance with the six core principles above. Code reviewers MUST verify adherence before approval.
- **Database Changes**: All schema migrations must include corresponding RLS policy definitions. Test policies before merging.
- **Deployments**: Edge Runtime functions MUST be tested locally with Edge mocks. Supabase RLS policies MUST be deployed and tested in staging before production.

## Governance

This Constitution supersedes all other development practices and architectural guidance for this project. Amendments require explicit documentation of rationale, affected principles, and migration steps for existing code. Version changes follow semantic versioning:
- **MAJOR**: Principle removals, backward-incompatible redefinitions, or new mandatory constraints that require code refactoring.
- **MINOR**: New principles added, existing principles expanded, or optional guidance layers introduced.
- **PATCH**: Wording clarifications, examples updated, typo corrections, or non-semantic refinements.

All team members MUST review this Constitution before committing code. Questions or clarifications should be raised in pull request comments with a reference to the relevant principle.
